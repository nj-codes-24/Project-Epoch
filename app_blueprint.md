# App Blueprint — The Student Builder Ecosystem
> A living document. Updated continuously as the vision evolves.

---

## Vision Statement

A platform built exclusively for student builders. Not a social network. Not a content feed. A place with real purpose at every stage of a builder's journey.

Students come for different reasons at different times — to learn what is happening at the frontier of tech, to share a frustration that has been bothering them, to find like-minded people who want to build something real, to solve a problem nobody else has tackled, to contribute to an open source project, or simply to help someone who is stuck.

Each of these is a genuine human motivation. The platform serves all of them without tricks, without manipulation, and without an algorithm designed to maximise time spent. People come back because the purpose keeps evolving as they do — from curious learner to contributor to builder to founder.

Every action a user takes is credited, verified, and converted into proof of work that travels outside the app and onto a resume. The reputation economy runs itself because helping others here genuinely helps yourself.

## The Five Pillars

### 1. The Validation Deck — Anonymous Demand Research

The entry point to the entire ecosystem. Before writing a single line of code, founders come here to find out if a real problem exists — without revealing what they are building.

The key design principle: **no ideas are posted publicly.** A founder never exposes their concept, their solution, or their product. They only post a behaviour-focused question about a human problem. This keeps the signal clean, eliminates IP anxiety, and makes the data more honest — voters are answering a question about their own life, not judging someone's pitch.

---

**How It Works**

The founder posts a vague, neutral question that probes demand without revealing intent.

Good question format: *"Would you use a tool that helps you [do something] when [situation]?"*

Examples:
- *"Would you use an app that automatically splits shared API costs between a group of student developers?"*
- *"When you can't find an auto, how many apps do you open before giving up?"*

The community votes using three tiers:
- `I would use this` — Active demand signal
- `Good to have` — Passive interest, not urgent
- `Do not need it` — Honest rejection

**Vote counts are hidden from the community while voting is open.** Every person votes in isolation, based purely on their own genuine experience — no anchoring, no social proof, no bandwagon effect. This produces cleaner data than most paid research tools.

Results are shown only to the question owner once the voting window closes.

---

**The Comment Section**

Comments are open for the full one-month lifecycle — including during the voting window. This is intentional. Votes give the founder a quantitative signal. Comments give qualitative depth. Both are valid and they serve different purposes.

The founder reads both and makes their own judgment call. The platform does not interpret the data for them. If votes are lukewarm but the comment section is highly active, that tells its own story — the problem resonates conversationally even if people wouldn't necessarily pay for a solution. The founder decides what that means for their build.

**Crediting commenters:** If a comment provides something genuinely useful, the question owner can mark it with one of four acknowledgement types. This is voluntary, not forced — a forced acknowledgement means nothing. But a genuine one is permanently logged on the commenter's Proof of Work profile and feeds into their badge progression. The four acknowledgement types and what triggers each one are defined below.

---

**The Acknowledgement Economy**

Badges do not appear next to a user's name. Every badge — from every feature — lives in a dedicated **Badge Showcase Page** on the user's profile. This is its own destination, separate from the name display, where the full collection is visible and grows as the user hits checkpoints across all features.

Each acknowledgement type has its own progression. The Badge Showcase shows every category as a card — current tier badge, category name, and contribution count toward the next checkpoint. Unearned categories are shown as locked, giving every user a complete map of what they have built and what they could build next.

All earned badges export to the resume PoW profile with their professional labels and verified contribution counts.

**The Full Lifecycle**

| Period | What happens |
|---|---|
| Week 1 | Question is live in the general feed. Voting and comments both open. |
| End of week 1 | Voting closes. Comments remain open. Question leaves the general feed but is searchable. |
| Weeks 2–4 | Question is read-only for the community. Owner reviews results. Comments still readable. |
| End of month | Everything deleted permanently. Blank canvas. No archive, no residue. |

Data expires by design. A validation from six months ago is a different market — new tools exist, student priorities shift, competitors may have launched. Old data is not shown as if it is still meaningful. The platform stays current by staying clean.

---

**Search Before You Post**

There is a universal search bar at the top of the app. Before posting a question, a founder will naturally search first — this is just how people use apps. The platform trusts users to behave this way without forcing a mandatory gate.

If an active or recently expired question matching their topic exists, they find it, read the results and comments, and decide if it is enough signal. If nothing comes up, or if the existing data feels too old or too different, they post a new question. No friction either way.

---

**Question Quality**

The platform nudges founders toward specific, behaviour-focused questions rather than vague or leading ones at the submission stage. Examples of weak questions are shown alongside strong ones so founders understand the difference before posting.

- **Too vague:** *"Would you use an app that saves you time?"* — meaningless signal
- **Leading:** *"Would you use an app that's 10x better than Uber?"* — everyone says yes
- **Strong:** *"When you can't book a ride on one app, do you immediately open another?"* — real behaviour, honest answer

The community itself acts as a natural filter over time — low-quality questions receive low engagement, which is visible signal in itself.

---

**What This Feature Is Not**

- It is not a pitch platform. No ideas are posted publicly.
- It is not an IP protection system. Questions are not ownable. Two people asking similar questions independently is not theft — it is a shared market insight.
- It is not a court. The platform does not adjudicate disputes. It provides honest signal and leaves interpretation to the founder.

The Validation Deck has one job: **tell you whether a problem is real before you invest time building a solution.**

### 2. The Discovery Feed — Showcase & Collaborate

A short-form video feed built specifically for startups, prototypes, and side hustles. The primary differentiator from LinkedIn, Product Hunt, and every other builder platform is the **Help Them** action — a live, bidirectional contribution system that turns passive viewers into active collaborators.

**A video cannot be posted without an active Help Board.** This is a hard gate. Every video on the feed is a genuine ask for help — not a broadcast, not an ad. This keeps the feed honest and purposeful.

---

**Onboarding — Skill Selection**

During onboarding, every user selects up to three skills from the following list. Their feed is curated to show only videos where the founder's Help Board tags match at least one of their selected skills.

Available skills:
`Code` · `Design` · `Writing` · `Research` · `Strategy` · `Analysis` · `Marketing` · `Sales` · `Finance` · `Legal` · `Product` · `Operations` · `Data` · `Video & Media` · `Community` · `Fundraising`

**Feed matching logic:** A user sees a video if at least one of their selected skills matches at least one of the Help Board's required skill tags. Partial match, not strict match — a user with `Code` and `Design` sees any video that needs either, not only videos that need both.

GitHub connection is also requested during onboarding via OAuth — redirected to GitHub's login page, permissions approved, returned to the app. The platform never handles passwords. GitHub connection is skippable but required to participate in code contribution tasks later.

---

**Core Features**

- **The Pitch** — Creators upload short videos explaining their business, showcasing a prototype, or demonstrating a new feature.
- **Standard Engagement** — Likes and comments on each pitch. The same four acknowledgement types from the Validation Deck apply here — video owners can mark comments with `Shaped My Thinking`, `Found The Gap`, `Sparked A Feature`, or `Clarified The Problem`, feeding directly into the commenter's Badge Showcase progression.
- **Direct Conversion Button** — Instantly opens the project's landing page, App Store link, or beta testing environment.
- **The "Help Them" Action** — Opens the project's Help Board. Every contribution made through this button is verified, logged, and permanently credited on both profiles.

---

**The Help Board**

Built by the founder at the time of posting. A curated list of specific, scoped tasks each tagged with the skill they require. Every task has:

- A clear, specific title — not "help with frontend" but "build a responsive login screen in React"
- A short description of what done looks like
- A skill tag from the sixteen-skill list
- Two states only — **Active** (open for contribution) and **Completed** (task is done)

The number of people working on a task is never shown. Showing a high number intimidates potential contributors into thinking there is no point. A task is either Active or Completed — nothing in between.

**The Help Board is strictly mandatory.** A video cannot exist on the platform without one. There is no fallback, no general signal, no workaround. This is the hard rule that keeps the feed free of ads and broadcasts permanently.

---

**The Contributor Flow**

1. Viewer watches the video, presses **Help Them**
2. The Help Board opens — tasks filtered by their skill match
3. Contributor claims a task — it is immediately Active and open for anyone to attempt
4. For **code tasks** — contributor works via GitHub and submits a Pull Request. Platform verifies the merged commit automatically.
5. For **non-code tasks** — contributor completes the work and submits the deliverable directly through the platform
6. Founder reviews the submission
7. If accepted — task marked Completed, no further claims accepted on that task, contribution logged on both profiles, badge progression updates, founder presses the acknowledge button (pre-prompted, name auto-filled, one tap only)
8. If revision needed — founder leaves a note, contributor resubmits

---

**The Three Contribution Types**

**1. Code Contributions — GitHub Integration**

- Founder posts the task with a direct link to the relevant GitHub repository or branch
- Contributor submits their code via a standard GitHub Pull Request — no workflow change
- Once the repository owner merges the commit, the platform backend automatically cross-checks the contributor's GitHub commit history against the linked repository
- If a match is found, both profiles are updated automatically

**Mandatory acknowledgement — Instant Lockout:** The moment a merged commit is verified, the founder's app locks immediately — regardless of what they are doing. Every element of the app blurs behind a frosted glass effect. The only visible element is a card in the centre of the screen:

> *"Someone helped you ship. Take 5 seconds to acknowledge them."*

A single green Acknowledge button. Nothing else is tappable. Nothing else is visible. The founder presses it and the app unlocks instantly. The acknowledgement is pre-prompted — the contributor's name and contribution details are auto-filled. The founder does nothing except press the button. No writing, no effort, no excuse not to.

The same instant lockout applies to non-code contributions once the founder has downloaded the final asset.

Users who skipped GitHub connection during onboarding are prompted to connect via OAuth the first time they attempt a code task.

---

**2. Visual Contributions — Design & Creative Assets**

- Contributor uploads their design or visual asset directly to the platform
- The work is visible to the founder as a **low resolution preview** — evaluable but not publishable or printable
- A canvas-rendered watermark is applied using two layers:
  - **Layer 1 — Identity Tile:** The founder's username, truncated email, and session timestamp tiled diagonally across the canvas at low opacity. Self-incriminating if screenshotted.
  - **Layer 2 — Intersecting Mesh:** A dense diagonal mesh grid using `globalCompositeOperation: multiply`, blending into the content to break OCR recognition
- Built-in chat allows iterative feedback and revision rounds
- **Mandatory acknowledgement — Gated Download:** The high-resolution, clean file is locked behind the **Give Acknowledgement** button. The founder cannot access it without crediting the contributor first.

---

**3. Written Contributions — Research, Copy & Strategy**

- Contributor uploads their document directly to the platform — not shared externally
- Rendered on an HTML5 canvas with the same two-layer watermark system (Identity Tile + Intersecting Mesh)
- Right-click is disabled on the canvas element
- The platform timestamps the submission against the specific project — this timestamp is the contributor's permanent proof regardless of what happens next
- **Mandatory acknowledgement — Gated Download:** The clean, unwatermarked document is locked until acknowledgement is given

**On protection limits:** No technical solution perfectly protects readable text. The canvas watermark, OCR resistance, and reputation deterrent together make honest behaviour easier than dishonest behaviour. That is the realistic goal.

---

**Founder Shoutout**

When a contribution is accepted, the founder is prompted to post a one-tap public acknowledgement tagging the contributor. Costs the founder nothing. Means a great deal to the contributor and is visible on both profiles permanently.

### 3. The Problem Board — Idea Bounties & Matchmaking

Not everyone has an idea, but many have the skills and the desire to build something useful. This space flips the script — focusing on **frustrations** instead of solutions. Think of it as a continuously running, community-judged hackathon with no organiser, no start date, and no end date.

---

**The Toggle — Two Views in One Feature**

The Problem Board has a toggle switch at the top with two states:

**Community Problem Board** — the default view. A live feed of frustrations posted by users, refreshing as new entries come in. Anyone can browse, vote, comment, or press `I Want to Build This`.

**Problems You Are Solving** — your personal workspace view. Shows a panel for every problem you have pressed `I Want to Build This` on. Each panel shows the problem statement title, how many teams are currently active, and a live activity indicator. Press any panel to enter that problem's Room. This toggle handles multiple simultaneous builds cleanly — five problems means five panels, all independent.

---

**Friction Posting**

Users post real-world frustrations they want solved. No solution required — just the problem stated clearly.

Example: *"I hate managing multiple apps to book a cab. Someone build a parallel booker."*

The community votes using three tiers:
- `I agree` — This frustrates me too
- `Good to have` — Minor inconvenience
- `Do not need it` — Not a real problem

The same four acknowledgement types apply in the comment section — `Shaped My Thinking`, `Found The Gap`, `Sparked A Feature`, `Clarified The Problem` — feeding into the commenter's Badge Showcase progression.

---

**The "I Want to Build This" Action**

Pressing this button does two things simultaneously:
- Adds a panel to the user's **Problems You Are Solving** toggle view
- Takes them directly into that problem's **Problem Room**

---

**The Problem Room — Discord-Style Layout**

Each problem has its own dedicated Room. The layout mirrors Discord — left column navigation, right side content panel.

**Left column structure (top to bottom):**

```
📌 Completed Products
   └─ Submission cards posted by teams who have finished
      (empty until someone ships — shows "No solutions yet")

💬 Main Community Chat
   └─ Open to everyone, not just builders

🔒 Your Team Chat
   └─ Only visible if you are in a team
      Appears automatically when you join or form one
      Never requires hunting through the teams list

👥 Teams (n)
   ├─ Team names listed as collapsible entries
   └─ Press any team to view their public profile

🧍 Solo Builders (n)
   └─ People building alone, with skill tags visible

🤝 Looking to Team Up (n)
   └─ People without a team actively seeking one
      Skill tags visible, Team Up request sendable from here
```

Right side — whatever is pressed on the left opens here.

---

**Teaming Up — Clash of Clans Mechanic**

From either the Solo Builders or Looking to Team Up tabs, any user can send a **Team Up request** to anyone else. The recipient gets a notification, reviews the sender's skill tags and PoW profile, and accepts or declines.

- **No team size cap.** Teams can be as large as they organically grow.
- **Solo building is valid.** No team requirement. Some of the best products are built alone.
- **If nobody accepts your request** — you remain in the Looking to Team Up tab and keep trying, or build solo.

---

**Inactive Member Credit Protection**

When a team is ready to submit, each member must attach evidence of their specific contribution before the submission is posted:
- Code role → merged PR link
- Design role → uploaded final asset
- Research or Writing role → uploaded document

The team lead reviews and confirms each member's evidence. Only confirmed contributors are credited on their PoW profile. A member who declared a role and submitted no evidence receives no credit, no badge progress, and no PoW entry. The platform does not spy on activity — the evidence speaks for itself.

---

**GitHub as the Progress Layer**

When a team forms, the first action in their private team channel is to post their GitHub repository link. From that point, commit activity is auto-populated into the team channel — commit messages, PRs, and merges appear automatically. Progress is visible without anyone needing to manually update. The Teams tab shows a live commit activity indicator next to each team so the broader community can see who is actively building.

For teams working on non-code products (design-only, research-only), team chat activity is used as the progress signal instead of GitHub commits.

---

**The Completed Products Tab**

A valid submission requires **both** of the following — no exceptions:
- A working prototype that can be tried
- A demo video showing the product being used in real time — not a mockup, not slides

Submissions are posted directly to the Completed Products tab at the top of the left column — never in the main chat. They are permanent cards, always pinned. The main community chat never gets flooded and completions are never buried.

The problem poster's vote counts equally with everyone else's — no disqualification regardless of whether they are on a competing team.

---

**The Event Lifecycle — How a Problem Gets Resolved**

**Phase 1 — Building**

Teams build. The strike system runs weekly in the background monitoring every active team independently.

Every week the platform checks each team:
- **Team committed to GitHub this week (or sent messages for non-code teams)** → no action, they are active
- **Team had zero activity this week** → strike message sent privately to that team:
  > *"Is your team still building, or have you folded?"*

| Strike | Response | Commits | Result |
|---|---|---|---|
| Strike 1 | Still Building | None | Still active |
| Strike 2 | Still Building | None | Automatically marked Folded |
| Any strike | We've Folded | — | Immediately marked Folded |
| Any strike | No response | — | Immediately marked Folded |

Active teams are never touched by the strike system. An active team building for six months stays active for six months. No event ever closes while at least one team is genuinely building.

**Phase 2 — The 2 Week Countdown**

The moment the **first valid submission** is posted to Completed Products, a 2 week countdown begins publicly. A banner appears in the Problem Room:

> *"First solution submitted. Voting opens in 14 days."*

During these 2 weeks:
- Other teams can still submit valid prototypes
- The strike system continues running in parallel
- No new submissions are accepted once the 2 weeks end

**Phase 3 — Voting**

When the countdown ends, voting opens with all valid submissions. Competing team members cannot vote on any submission — only non-builders vote. The problem poster votes with equal weight to everyone else.

Voting outcomes by submission count:

| Submissions | Voting format | Result |
|---|---|---|
| 3 or more | Top 3 go to vote | 1st, 2nd, 3rd place awarded |
| 2 | Both go to vote | 1st and 2nd place awarded |
| 1 | Single vote | Solved or Not Solved |
| 0 | No vote | Event closed, marked Unbuilt |

**If the first submission never happens** — the strike system eventually folds all inactive teams. Once all teams have either submitted or been marked folded with zero valid submissions, the event closes and is marked **Unbuilt.** The post remains permanently visible as a signal that this problem exists and has not been solved yet.

---

**Rewards by Placement**

**Problem Poster** — `Problem Originator` badge progresses by 1. Their post gets a permanent Solved tag with all solutions linked beneath it. PoW card shows the problem and the products it produced.

**1st Place Team** — `Bounty Hunter` badge progresses by 1. PoW card shows **1st Place** against the specific problem permanently. All members credited individually with their declared role and evidence.

**2nd and 3rd Place Teams** — `Bounty Hunter` badge progresses by 1. Shipping a working product is the achievement regardless of ranking. PoW card shows **2nd Place** or **3rd Place** permanently. The distinction lives as a label, not in the badge count.

**Community Voters** — voters who voted for the eventual 1st place winner earn a mark for that event feeding into the `Community Judge` badge progression. Rewards accurate judgment, not just participation.

### 4. The Knowledge Hub — Curated Research

Staying updated on cutting-edge tech is exhausting. This feature removes the friction entirely. Every paper is fetched, summarised, and made actionable by AI — delivered to the user on a clean, digestible card. No hunting, no jargon, no overwhelm.

This feature is entirely AI-driven. No community curation, no manual editorial work. The pipeline runs automatically, the content stays fresh, and the user just reads.

---

**The Data Sources**

Two free public APIs power the paper feed:

- **ArXiv** — maintained by Cornell University. Every major AI, ML, and computer science paper is published here first, before any journal. Free API.
- **Semantic Scholar** — built by the Allen Institute for AI. 200 million papers, ranked by citation velocity and community traction. Free API.

Both are queried daily. Results are deduplicated and filtered to the ten defined categories.

---

**The Ten Categories**

The Knowledge Hub is organised into ten topic shelves — one row per category, top 5 papers per row:

`Artificial Intelligence` · `Machine Learning` · `Data Science` · `Web Development` · `Mobile Development` · `Cybersecurity` · `Blockchain & Web3` · `Computer Vision` · `Natural Language Processing` · `Robotics & Hardware`

---

**The Daily Pipeline**

```
Scheduled job runs every 24 hours at low traffic time
        ↓
Query Semantic Scholar + ArXiv for top 20 papers
per category, ranked by citation velocity
        ↓
Deduplicate against all previously processed papers
Only genuinely new entries are processed
        ↓
New papers sent to Gemini API with structured prompt:
— Plain language summary
— Why a builder should care right now
— 2-3 open source tools or repos that implement
  this concept today, free to use
— Visual formatting instructions
        ↓
Output stored in database as a structured card
        ↓
Card goes live in the Knowledge Hub feed
Topic tags extracted and stored for contextual use
```

**Always fresh per user:** The platform tracks which papers each user has already opened via a personal seen list. When the Knowledge Hub loads, it surfaces the top 5 unseen papers per category. If today's top 5 are all already seen, it moves to the next best unseen ones. Every session feels new.

---

**The Monthly Database Refresh**

A separate monthly job keeps the contextual database current for the Knowledge Panel integration:

- **Tool version check** — queries GitHub API for latest release on every stored open source tool. Updates version info automatically. Flags and removes deprecated tools with no commits in over a year.
- **Paper relevance check** — queries Semantic Scholar for citation velocity on stored papers. Papers superseded by newer work get flagged. Newer developments are pulled in.
- **Contextual tag rebuild** — topic tags used for Knowledge Panel matching are regenerated monthly based on the refreshed database.

This ensures the Knowledge Panel always surfaces current tools and current research — never six-month-old recommendations.

---

**The User Experience — Netflix-Style Layout**

The Knowledge Hub feed is a grid of category shelves. Each row is a category label. Each card in the row is one of the top 5 papers for that category. Scroll horizontally within a category, vertically to move to the next one. Clean, browsable, never overwhelming.

Each card shows:
- Paper title in plain English
- One-line summary of what it means for a builder
- Gemini-generated visual breakdown — diagrams, highlights, plain language
- 2-3 open source tools with direct links
- **Save button** — stores the paper to the user's Saved section on their profile
- **Share button** — sends the card directly to a DM or into a Team Channel inside a Problem Room

The Saved section on the user's profile has two tabs:
- **Papers** — saved research breakdowns
- **Tools** — individually saved open source tools

Tools can be saved independently from their parent paper. If a paper surfaces three tools and the user only wants one, they save just that tool.

---

**The Knowledge Panel — Problem Board Integration**

Every Problem Room has a **Knowledge Panel** pinned at the very top of the left column — above Completed Products, above everything else.

The panel is **read-only and identical for all users** inside that room. No interaction, no personalisation, no typing. Every person in the room sees the exact same five papers and tools, pulled automatically from the topic tags of the problem statement when the room is created.

If a displayed paper or tool has been saved by the user previously, a **Saved mark** appears on that card — a passive visual indicator that they have already read it. It does not change what is shown, only signals familiarity.

The Knowledge Panel updates automatically when the monthly database refresh runs, ensuring tools and papers shown are always the most current available for that topic.

**Integration with the Validation Deck:** When a validation question is posted, the platform detects the topic and surfaces relevant Knowledge Hub cards directly below the question — same read-only, same logic, same five papers and tools. The founder validates demand and discovers the technical landscape simultaneously.

**The Search Bar**

A search bar sits at the very top of the Knowledge Hub. The user types their project idea or topic and gets results in two independent sections:

**Research Papers** — queries the existing processed database only. Returns the top 5 to 10 most relevant papers already in the database for that topic. If the topic does not exist in the database yet, the user sees a transparent message:

> *"We don't have research papers on this topic yet. Our database updates daily — check back soon."*

No on-demand processing, no fake completeness. Honest and simple. As the database grows over time this empty state becomes increasingly rare.

**Open Source Tools** — a live GitHub API call for any search term, free and instant. Returns the top repositories ranked by stars, forks, and recent commit activity. No AI processing needed — the quality signal is built into GitHub's own ranking. Each tool result shows the repository name, one-line description, star count, last commit date, a direct link, and a Save button.

The search bar behaviour by scenario:

| User searches | Source | Result |
|---|---|---|
| Topic with papers in database | Existing database | Top 5–10 paper cards shown |
| Topic with no papers in database | — | Honest empty state message |
| Open source tools for any topic | Live GitHub API | Top repos by stars, free and instant |

## The Reputation Economy — Why People Come Back

The platform is built on two of the strongest student motivations: **ego (the flex)** and **career survival (the resume)**. Every action on the platform feeds into a reputation system that rewards participation with things that matter outside the app.

### 1. The Verified Proof of Work (PoW)

The most important career feature on the platform. The gap between "I helped a startup" and "here is verifiable proof of what I did" is where most student portfolios fall apart. PoW closes that gap.

**How it works:**
- Every contribution — a comment that shaped a product, a code commit, a logo, a market research response — is automatically logged on the user's PoW profile tab.
- The log is immutable. Founders cannot retroactively remove a contributor's credit.

**The Output — Case Study Cards:**
Instead of a wall of stats, each contribution is rendered as a compact **Case Study Card** showing:
- The project name and a one-line description
- The contributor's specific role and what they did
- The outcome (e.g., "MVP launched," "200 beta users acquired")
- One human endorsement from the founder or a teammate (see Endorsements below)

Stack these cards and the result is a narrative portfolio — something that reads fast, travels well, and tells a story rather than just displaying a number.

**The Shareable Link:**
Each user gets a public PoW URL — a clean, interactive resume they can drop into a LinkedIn post, a job application, or a cold email to a recruiter.

Displayed metrics include:
- *"Contributed to 4 launched MVPs"*
- *"Ranked top 5% in Community Problem Solving"*
- *"Validated 12 product pitches"*
- *"Day One Believer on 3 products that launched"*

**Dual Naming for Exported Labels:**
Badge names are fun and personality-driven inside the app, but export-ready for professional contexts. Every badge has two names:

| In-App Name | Exported / Resume Label |
|---|---|
| Jugaad Master | Resource Optimisation Contributor |
| Bounty Hunter | Product Problem Solver |
| The Architect | Technical Strategy Contributor |
| The Fixer | Debugging & Design Consultant |
| The Visionary | Validated Idea Originator |
| Day One Believer | Early Market Validation Contributor |

### 3. Endorsements With Context

A one-click LinkedIn endorsement is meaningless. On this platform, endorsements are contextual and attached permanently to a contributor's Case Study Card.

When a founder accepts a "Help Them" contribution, they can write a short endorsement that travels with that card forever:

> *"Fixed our auth flow in 48 hours before our college demo day."*

This turns the community into a **trust network** — verifiable, specific, and impossible to fake.

## The Credit Philosophy

The most important cultural principle of the platform: **every person who touches a project gets credited, permanently and specifically.**

This covers:
- The person who posted the original problem
- The person who posted the idea

## Background Jobs (Trigger.dev)

### Job 1 — Daily Paper Pipeline
**ID:** `daily-paper-pipeline`
**Schedule:** Every day at 02:00 UTC (low traffic window)

**Steps — runs for each of the 10 categories in sequence:**

```
1. Query Semantic Scholar
   — endpoint: /graph/v1/paper/search
   — params: fields=title,abstract,year,citationCount,externalIds
   — sort by: citationVelocity, last 30 days
   — fetch top 20 results

2. Query ArXiv
   — endpoint: http://export.arxiv.org/api/query
   — params: category, sortBy=submittedDate, max_results=20
   — fetch top 20 results from last 7 days

3. Deduplicate
   — Check each result against papers table (arxiv_id OR semantic_id match)
   — Only keep genuinely new papers

4. For each new paper — call Gemini 1.5 Flash
   — Send title + abstract with structured prompt (see Gemini Prompt below)
   — Parse JSON response
   — Insert row into papers table
   — For each tool in response: insert row into tools table (skip if github_url already exists)

5. Rate limit guards
   — Semantic Scholar: 1 request/second max
   — Gemini: 10 requests/minute max (free tier: 15 RPM ceiling)
   — ArXiv: 1 request/3 seconds (be a good citizen)

6. Error handling
   — Gemini failure on a paper: retry up to 3 times with exponential backoff
   — If all 3 retries fail: log and skip that paper (do not block the pipeline)
   — API failures (ArXiv/Semantic Scholar): log and surface alert, retry next day
```

---

### Gemini Prompt Template (Daily Pipeline)

Send as a single user message. System instruction tells the model to return only JSON.

**System instruction:**
```
You are a research assistant for student software builders. Always respond with valid JSON only — no markdown, no preamble, no explanation. If you are not confident a GitHub repository exists, omit it rather than guess.
```

**User message:**
```
Summarize this research paper for a student builder audience.

Title: {{title}}
Abstract: {{abstract}}

Return this exact JSON:
{
  "plain_title": "...",        // Title rewritten in plain English, max 12 words
  "summary": "...",            // What this paper does, 2 sentences, no jargon
  "why_it_matters": "...",     // Why a builder should care right now, 1 sentence
  "topic_tags": ["...", "..."],// 3 to 5 lowercase topic keywords
  "tools": [
    {
      "name": "...",
      "github_url": "https://github.com/...",
      "description": "..."     // One line description of what the tool does
    }
  ]                            // 2 to 3 real open source GitHub repos implementing this concept. Omit any you are not confident exist.
}
```

---

### Job 2 — Monthly Database Refresh
**ID:** `monthly-db-refresh`
**Schedule:** 1st of every month at 03:00 UTC

**Steps:**

```
1. Tool version check (for all tools where is_deprecated = false)
   — Query GitHub API: GET /repos/{owner}/{repo} for each tool
   — Update: stars, forks, last_commit_at
   — If last_commit_at is older than 12 months: set is_deprecated = true

2. Paper citation refresh
   — Query Semantic Scholar for updated citation count on all active papers
   — Update citation_count in papers table
   — If a paper has been superseded by a newer version (detected via Semantic Scholar fields):
     set is_active = false on the older entry

3. Topic tag rebuild
   — For all papers where topic_tags is empty or NULL: re-run Gemini tag extraction
   — Update topic_tags in place
```

## "Always Fresh" Feed Logic

The Knowledge Hub shows the top 5 **unseen** papers per category for each user.

**Query logic (run on feed load):**
```sql
SELECT p.*
FROM papers p
WHERE p.category = $1
  AND p.is_active = true
  AND p.id NOT IN (
    SELECT paper_id FROM user_seen_papers WHERE user_id = $2
  )
ORDER BY p.citation_count DESC, p.processed_at DESC
LIMIT 5;
```

If fewer than 5 unseen papers exist in a category → fill remaining slots with the most recent active papers from that category (seen or not), so the shelf is never empty.

A paper is marked seen when the user **opens** it (taps the card) — not on scroll-past.
